  1. Open or import the project folder:

If Godot asks for the main scene, choose Scenes/Main.tscn. If C# changes are not picked up, use the MSBuild panel and build once.

Viewer Controls

  • S: collapse or expand the scenario launcher
  • Space or P: pause or resume
  • R: reset the current seed
  • N: create a new seed and reset
  • F: fit the view to the world
  • G: follow the selected creature at a closer map zoom
  • B: show or hide biome colors
  • C: cycle creature color mode
  • + / -: change simulation speed
  • Arrow keys: pan the map camera
  • Shift + arrow keys: pan faster
  • Mouse wheel: zoom at cursor
  • Right or middle mouse drag: pan
  • Left click: select a creature

Scenario Launcher

The launcher panel lets you edit all shared scenario settings, then launch them visually in Godot or run a headless CLI experiment from the same values.

  • Launch/Restart: restarts the Godot viewer using the edited settings
  • Load: loads a scenario JSON file into the editor and restarts the viewer
  • Save / Save As: writes the edited scenario as CLI-compatible JSON
  • Collapse / Expand: shrinks the launcher to a small strip and moves realtime stats into the left column
  • Run CLI: saves the edited scenario to out/godot_cli_scenario.json, then runs Lineage.Cli with the requested tick count, stats CSV, derived CSV sidecars, HTML report path, and final snapshot path
  • Write Current Report: writes an HTML report with graphs and behavior assays from the simulation currently running in Godot, without rerunning the CLI
  • Open Report: opens the last successful CLI or viewer HTML report in your default browser
  • Snapshot JSON: sets where a Godot-triggered CLI run writes its final simulation snapshot
  • Checkpoint interval: writes loadable snapshot checkpoints every N ticks; use 0 to disable
  • Checkpoint folder: sets where periodic checkpoint JSON files are written
  • Load Snapshot File: loads a saved simulation snapshot into Godot for inspection
  • Load Checkpoint File: opens the checkpoint folder so you can choose a saved checkpoint
  • Load Last Snapshot: loads the final snapshot from the most recent successful Godot-triggered CLI run
  • Load Latest Checkpoint: loads the most recent checkpoint from the last successful Godot-triggered CLI run

Run The Local CLI Runner

The runner is a local web dashboard for starting and monitoring one or more Lineage.Cli simulations without using the Godot launcher or managing terminal windows manually.

Start it from the repository root:

dotnet run --project .\src\Lineage.Runner\Lineage.Runner.csproj --urls http://localhost:5088

Then open:

http://localhost:5088

Current Runner Features

  • Choose a scenario, tick count, optional seed, optional checkpoint interval, and extinction stop behavior
  • Expand grouped scenario options before launch, with tabs and field metadata shared from Lineage.Core
  • Search scenario options, switch between basic and all fields, reset a group or the whole loaded scenario, and see changed fields highlighted
  • Save edited launch settings as launcher-managed scenarios under scenarios\user
  • Archive launcher-managed saved scenarios while keeping built-in project scenarios protected from dashboard deletion
  • Start multiple CLI simulations as managed runs
  • Watch progress, current tick, creature count, egg count, species cluster count, generation, births, deaths, and checkpoint count
  • Request Stop, Checkpoint, or Checkpoint + Stop for an active run
  • View completed run reports from the run table
  • Search, sort, and filter runs by status or scenario
  • See a concise failure reason on failed, lost, or unknown runs when stderr or runner diagnostics are available
  • See the seed, brain/starter kind, world size, and resource density used on each run row
  • Clone settings from a running or finished run back into the launch form for quick experiment variants
  • Rerun a completed, failed, or lost row after confirming that the previous run artifacts will be replaced
  • Continue a completed, failed, or stopped run in the same row and output folder from its latest checkpoint, final snapshot, or a selected checkpoint in run details
  • Open compact run details with command line, paths, resolved scenario path, artifact/checkpoint picker, and log tails
  • Export selected run summaries and scenario identity as Markdown for cross-run review or Codex analysis
  • Rename runs and delete completed run records individually or in bulk

Each dashboard-launched run gets its own folder under out/runs/<run-id>. The run id ends with the CLI process id. The folder contains a manifest, resolved scenario JSON, stdout/stderr logs, status.json, control.json, stats CSVs, the HTML report, the final snapshot, and checkpoint files.

When scenario options are edited in the runner, the dashboard writes a generated launch scenario under out/runs/_launch_scenarios, passes that to Lineage.Cli, and still records the final CLI-resolved resolved_scenario.json beside the run manifest.

Saved scenario variants created from the runner live under scenarios\user. The runner can archive those saved variants to out/scenario-trash, but project scenarios directly under scenarios are treated as protected presets and are not deleteable from the dashboard.

Runner Notes

  • The runner can also be started directly from its built executable after a build: .\src\Lineage.Runner\bin\Debug\net8.0\Lineage.Runner.exe --urls http://localhost:5088
  • If port 5088 is already in use, choose another local port in the --urls value.
  • The runner reloads manifests on startup and reattaches to live CLI processes by process id, so status and control actions can continue after a runner restart.
  • If a manifest says a run was active but the recorded CLI process is gone, the runner marks it lost instead of pretending it is still controllable.
  • If you need to rebuild while the runner executable is active, stop the Lineage.Runner process first so Windows can replace the executable.

Current pressure presets:

  • .\scenarios\gentle-foraging.json: abundant food, smaller void border, lower basal/movement pressure, and moderate reproduction delay
  • .\scenarios\balanced-foraging.json: baseline food, void border, energy pressure, longer maturity, and higher reproduction threshold
  • .\scenarios\harsh-foraging.json: sparse food, wider void border, slower regrowth, and higher basal/movement pressure
  • .\scenarios\scavenger-pressure.json: lower plant availability, wider void, richer slower-decaying carcasses, and balanced-style combat/egg rules for testing scavenging pressure
  • .\scenarios\omnivore-pressure.json: middle pressure between scavenging and predation, with richer carcasses and modest combat pressure while keeping the seed-forager starter brain
  • .\scenarios\predation-pressure.json: lower plant reliability, richer persistent meat, slower reproduction, a forager-predator starter brain, and moderated costly bites for testing active predation pressure

Large worlds need proportionally more resources or longer-lived creatures. The Godot viewer now treats the center view as a map camera: use the mouse wheel to zoom, arrow keys or right/middle mouse drag to pan, F to return to the full-world view, and G to follow a selected creature. The scale bar in the lower-right of the map shows the current world-unit distance. Biomes are generated from the scenario seed; richer biomes attract more initial and relocated resource patches. Scenario-backed resource clustering can make plants form local food patches instead of evenly dusting the world. The resource void border is a no-plant/no-regrowth edge zone. Biomes also apply indirect movement and basal metabolism cost multipliers, so sparse and barren ground can be harder to cross without giving the brain perfect biome knowledge. Movement cost is also nonlinear by actual speed, so slow cruising is cheap and high-speed travel becomes increasingly expensive. Press B to toggle the biome and void underlay.

Creatures do not get direct biome fertility or void-border knowledge. Food direction and proximity are only exposed for edible resources inside the creature's current vision cone, while the brain receives visible food density as a local crowding cue. Selecting a creature draws that vision cone. Vision range and vision angle are heritable traits, and both have upkeep costs so maximum-range 360-degree vision is not free.

Food currently has two kinds: plants and meat. Plants regrow and relocate when fully depleted; meat is left by dead creatures, decays, emits local scent, and disappears. Meat scent can be detected farther away than exact vision, but it only gives a fuzzy density and forward/right gradient. A creature still needs visible food inside its cone for exact target direction and proximity. The current presets use scent out to 2x effective vision range, 60 kcal as the full-strength scent scale, and 1.0 scent-density saturation. Eggs are also edible meat-like targets: they can be seen only inside the vision cone, use meat digestion efficiency, and count as egg predation deaths if fully consumed. Living creatures can also be sensed inside the vision cone as neutral creature cues: density, proximity, direction, relative body size, relative speed, approach rate, and facing alignment. A creature with attack intent can bite another creature at body-contact range; biting costs energy, deals health damage, and injury deaths become meat. Bite strength and damage resistance are heritable, growth-scaled traits: juveniles hit less hard and take more damage, while stronger or tougher adults pay more upkeep. A heritable diet value controls digestion: plant specialists get strong plant calories but weak meat calories, meat specialists mirror that, and omnivores sit in the middle. Food is not instantly converted to energy: eating fills plant/meat gut stores, then digestion releases usable energy over time. Fast digestion has upkeep, slow digestion can leave the gut full, and larger gut capacity also costs upkeep. The current starter scenarios use a 55 kcal gut, 5 kcal/s digestion, 0.0008 gut-capacity upkeep, and 0.014 digestion-rate upkeep. The balanced starter begins near herbivore behavior, with about 92% plant digestion and 28% meat digestion. Neural brains receive both a generic diet-weighted best-food cue and separate plant/meat density, direction, proximity, meat-scent cues, and neutral visible-creature cues. This keeps the information local while allowing lineages to evolve different foraging responses as their digestion shifts.

The live HUD and HTML reports include foraging diagnostics: share seeing food, share touching food, share eating on the sampled tick, average visible food density, meat-scent detection and density, raw calories eaten per second, raw intake split by plants, passive carcasses, eggs, and attacker-credited fresh kills, calories digested per second, digested energy split by plant and meat source, average gut fullness and gut composition, plant/meat resource calories, egg predation deaths, visible creature density, creature-detection share, attacking creatures, attack damage per second, fresh-kill/meat intake shares, meat-derived energy share, attacker versus non-attacker diet/bite/resistance averages, average time since last meal, movement distance per second, distance since last meal, calories per distance, calories per food-vision event, and average effective vision range/angle. HTML reports also include inline SVG graphs for population, eggs, resource calories, foraging signals, search efficiency, food-source intake, predation diagnostics, digested energy source, digestion, and combat pressure, plus behavior assays that replay fixed food, visible-creature, small/large-creature risk, and reproduction cues through the living neural brains. The same probes are also grouped by top living founder lineage so minority ecotypes are easier to spot.

Resource and creature rendering are zoom-aware. Close views draw individual entities. Wide or crowded map views switch to density cells so very large worlds do not try to paint every plant patch or creature body.

The HUD includes performance telemetry: TPS is actual simulation ticks processed per real second, Frame is average frame time, v is visible/estimated entities in the current view, and d is the number of individual entities or density cells drawn.

Useful Options

  • --scenario <path>: load a scenario JSON file
  • --save-scenario <path>: save the resolved scenario before running
  • --ticks <n>: number of simulation ticks to run
  • --seed <n>: override deterministic scenario seed
  • --pipeline <neural|simple>: override controller pipeline
  • --creatures <n>: override initial creature count
  • --resources-per-million-area <n>: override initial resource density; the actual patch count scales with world area
  • --resources <n>: legacy absolute resource count override; converted to density for the resolved world
  • --snapshot-interval <n>: override stats snapshot interval
  • --output <path>: stats CSV output path
  • --lineage-output <path>: lineage CSV output path
  • --traits-output <path>: final trait summary CSV output path
  • --species-output <path>: living species cluster summary CSV output path
  • --species-trends-output <path>: species cluster trend CSV output path
  • --founders-output <path>: founder summary CSV output path
  • --generations-output <path>: generation survival CSV output path
  • --lineage-trends-output <path>: founder lineage trend CSV output path
  • --report <path>: HTML run report output path
  • --save-snapshot <path>: final simulation snapshot JSON path that can be loaded back into Godot
  • --checkpoint-interval <n>: save loadable checkpoint snapshots every N ticks
  • --checkpoint-dir <dir>: directory for checkpoint JSON files; defaults near the snapshot, report, or stats output
  • --batch-report <path>: HTML comparison report output path; uses gentle/balanced/harsh presets when no batch scenarios are listed
  • --probe: run lightweight tuning probes without per-run reports, snapshots, or lineage CSV suites
  • --probe-scenario <path>: add a scenario to a lightweight probe; can repeat
  • --probe-seeds <a,b,c>: comma-separated seed overrides for probe runs
  • --probe-output <path>: compact probe CSV output path
  • --probe-report <path>: compact probe HTML output path
  • --probe-snapshot-interval <n>: stats interval for probe runs; defaults to 100
  • --probe-stop-on-extinction: stop a probe run early if all creatures die
  • --probe-max-population <n>: stop a probe run early if population exceeds a safety cap
  • --batch-scenario <path>: add a scenario to a comparison run; repeat for multiple scenarios
  • --batch-output-dir <dir>: per-run batch CSV/report output directory
  • --no-output: run without writing CSV files

Current Notes

  • The viewer and CLI share the same default scenario factory.
  • The simulation already has mutation, inheritance, reproduction, death, and lineage records.
  • Reproduction lays eggs. Mature creatures first build a reproductive reserve from surplus energy, then neural brains can decide when to deposit a completed egg with the mutated offspring genome and brain. Eggs have investment-derived health, can take environmental exposure damage in poor locations, can be eaten as meat-like food by other creatures, incubate for a heritable duration, and hatch into lineage-tracked creatures if viable.
  • Juveniles grow toward adult body size, speed, eating capacity, gut capacity, and digestion rate, and cannot reproduce before maturity.
  • Food search is local and imperfect: creatures target only visible food inside their vision cone, receive visible food density rather than map fertility, and smell nearby meat as a fuzzy gradient rather than an exact target.
  • Dead creatures leave meat resources. Living creatures can now see other living creatures through neutral creature cues and bite one another at close range when their brain outputs attack intent. Bite strength and damage resistance are heritable and costly. Diet adaptation, gut capacity, and digestion rate are heritable, so lineages can evolve from mostly herbivory toward omnivory or meat specialization when the scenario makes that viable.
  • Evolutionary pressure is configurable through biome-shaped food density, resource clustering, no-plant void borders, regrowth, death-meat output, meat decay, meat scent, initial brain mode, bite strength, damage resistance, bite damage/cost/reach, combat-trait upkeep, metabolism, movement cost, shared trait upkeep, vision range/angle upkeep, gut-capacity and digestion-rate upkeep, optional egg upkeep, egg exposure damage, egg production rate, offspring investment, egg incubation, maturity age, reproduction threshold, and body-size upkeep.
  • Offspring investment sets egg sturdiness and hatchling birth quality; explicit fat, strength, and clutches can build on this later.
  • The latest pressure tuning added an omnivore preset and softened predation-pressure. At 20k ticks, omnivore-pressure settles near 300 creatures with meat contributing about 7% of energy, while predation-pressure settles near 150 creatures with sustained fresh-kill intake and meat contributing about 12% of energy.
  • Local carcass/meat scent and neutral creature perception are now in place. The next simulation work should improve evolved behavior around risk, avoidance, and attack decisions rather than only tuning food and damage numbers.
